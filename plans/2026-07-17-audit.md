# Gradle Shardwise Plugin — End-to-End Audit & Fix Program

> **Mode:** autonomous. Single source of truth for any session that resumes this work.
> **Authority model:** full-autonomy. Owner approved executable scope; gate-clean only.

---

## 1. Goal

Audit `gradle-test-shard-plugin` end-to-end — code, docs, structure, tests, CI/CD, DX — across **at least 10 review iterations** using multiple specialized lenses. Every confirmed finding is fixed. Each fix batch is reviewed by an expert lens focused on the changed symbols. Final deliverable: green gate, no residual BLOCKER findings, autonomy-resumable plan.

## 2. Non-Goals

- No new features.
- No public DSL breakage (changes to `ShardwiseExtension`/`ShardwisePlugin` API surface). If a finding requires it, escalate to owner.
- No dependency version bumps beyond what a security/correctness finding forces.
- No removal of `e2e/` or GitHub Actions triggers.

## 6. Per-Iteration Loop (5 steps, applied to every row above)

1. **Lens sweep** (read-only). Scope = surface in row's "Surface" cell. Output = findings ledger with skill's finding-ID prefix (`GRADLE-*`, `TKIT-*`, `GA-*`, etc.), tier (BLOCKER / WARNING / INFO), file:line evidence.
2. **Triage.** BLOCKERs gate; WARNINGs batch; INFOs note. Owner escalation only for `public API break` / `credential surface` / `permission widening`.
3. **Fix batch.** 2–5 related findings, branch `audit/iter-N-<slug>`. Surfaces only touched files. Targeted `git add <paths>` (lesson 14). No `git add -A`. No force-push.
4. **Re-review.** Same lens, scoped to changed symbols. Reject batch if any BLOCKER is found.
5. **Integration.** Merge ff-only into main under lock; status commit on main; cycle continues.

## 7. Hard Rules Across Every Fix

- **`api/*.api` diff** → escalation to owner (binary compat).
- **Test floor** = current real green test count at program start. Cannot decrease (lesson 6, 26). Asserted in gate.
- **No** `skip`/`@Disabled`/deleted tests outside a dedicated test-removal step.
- **No** loosened assertions, no reference-identity replacing deep-equal (lesson 26).
- **No** secrets in diffs. **No** workflow permission widening.
- **No** untyped `!!` (`!!`-is Kotlin's `unsafe unwrap` — same blast radius as Rust's `unwrap()`); replace with `requireNotNull` + narrow type.
- **No** `System.getenv()`/`System.getProperty()` at config time → must use `ValueSource`.
- **No** `afterEvaluate { }`, no `project.container()`, no config-time `.get()` on `TaskProvider`.
- **Targeted** `git add <paths>` only. Abort if unrelated files are dirty.
- **`actionlint`** before every workflow-touching commit (CLAUDE.md invariant).

## 8. Gate (every commit, every iteration)

```bash
./gradlew detekt                                    # zero new findings vs iter baseline
./gradlew check                                     # detekt + unit + functionalTest + apiCheck
./gradlew apiCheck                                  # binary compat stable
./gradlew tasks.validatePlugins                     # failOnWarning → must be silent
actionlint .github/workflows/*.yml                  # zero findings
test-count-floor-check                             # current count ≥ baseline
```

If any gate command fails, iteration halts; do not advance; root-cause before re-gate.

## 9. Operating Model (per `autonomous-long-run` invariants)

- **Lock:** `audit.lock.d` with start-ts meta; stale TTL 45 min; release only via `rm -rf` after confirm (lesson 19).
- **Test-floor baseline:** captured at iter-1 start, recorded in `audit-baseline.json`; never decreases.
- **Status commits go on `main`** (lesson 8) so resume agents find them.
- **Push** is `git push origin main` (plain push, no `--ff-only` flag — lesson 18).
- **No implicit human-in-the-loop**; gate-green is the merge trigger.
- **Plan mutation** on resume — read `git status` first, reactivate codegraph if dropped (lesson 29).

## 10. Termination Conditions

- (a) **Zero** new findings in **2 consecutive full sweeps** across all lenses, OR
- (b) **12 iterations** reached with diminishing-returns report, OR
- (c) **Hard blocker** → convene the §14 expert council (3 voices: correctness / scope / alternative-substitution) to choose between (a) terminate the program, (b) substitute an in-scope alternative, or (c) escalate to owner. Council outcome is terminal for the run.

## 11. Reporting Contract

- **Mid-run** message ONLY on: BLOCKED that needs owner, escalated judgment, status milestone completion.
- **Before the final message:** update `CLAUDE.md` per §15 — bake new conventions, durable findings (antipatterns confirmed), and any lint/lint-config adoptions into the project-rules file. Audit lessons only persist if the file reflects them.
- **End-of-program** message: total iterations run, finding-count delta per lens, gate receipts, terminal `JUDGMENT: APPROVED ✅` or `JUDGMENT: ESCALATED ⚠️`.

## 12. Risks I Already See (provisional, will be confirmed or rejected by iter-1)

1. **No `gradle/libs.versions.toml`** — dependencies inlined in `build.gradle.kts`. Renovate groups gradle updates but there's no version catalog to graduate them into.
2. **`ShardBuildService.Params`** — needs full audit for Serializable correctness and `usesService` wiring.
3. **`ShardingWeights.parse`** — silently drops invalid lines (E-10 confirms this is INTENTIONAL but it's worth a test that asserts the silent-drop semantics, not only that valid lines pass).
4. **`scripts/mutation-test.sh` + `verify-test.sh` + `run-node-env-test.sh` + `glci-fetch-node-logs-test.sh`** — CI scripts have their own tests, which is good; need to verify they test what their names claim.
5. **`codeql.yml` is `workflow_dispatch` only** — intentional per comment, but: a check that *can never be green* is itself a smell; consider deleting or wiring back per the comment's escape hatch.
6. **`paths-ignore` excludes all `*.md`** in `ci.yml` — docs silently drift; iteration 4 surface.
7. **Detekt `style/ForbiddenComment` only forbids `FIXME`/`STOPSHIP`, not `TODO`** — fine, but verify no `TODO` is left in committed code.
8. **e2e/consumer/build.gradle.kts version is grep-asserted at publish time** — drift check; verify it matches root `version` in CI.

## 14. Expert Council Procedure for Hard Blockers

When a finding (or stack of related findings) surfaces a real hard blocker — a tension the autonomous run cannot resolve on its own — the orchestrator does NOT silently absorb scope, NOR silently terminate. It convenes a **3-voice expert council** in a single ad-hoc turn, with fresh context per voice, and binds the outcome.

**Voices** (parallel fan-out, mid-tier model):
| # | Voice | Angle | Reads |
|---|-------|-------|-------|
| A | **Correctness Reviewer** | Is the blocker a real correctness gap (will it bite in production)? | Source + reproducer + skill rubric for the lens |
| B | **Scope Reviewer** | Was this in the original owner-approved scope, or does fixing it require scope expansion? | Plan sections 1, 2, and the iteration 5 risk ledger |
| C | **Alternative-Substitution Reviewer** | Can the audit goal still be met with a different in-scope substitute? | Plan sections 1, 2, and the broader repo capabilities |

**Ruling options** (synthesis writes ONE):
| Option | Meaning |
|--------|---------|
| **Terminate** | The blocker is unrecoverable in the current scope; end the program; surface to owner |
| **Substitute** | Document the substitution (what we replace, with what, why still in scope), continue |
| **Escalate** | Option neither substitute nor terminate fits; hand to owner with three-voice summary |

**Rules:**
- Convergence early-exit: stop when voices stop producing new substantive objections.
- Skeptic holds binding veto: if any voice names a definite risk the others ignore, that risk is added to the new finding list before any continue.
- The chosen ruling is appended to the audit plan (this file, §14 below this section) with timestamp, voice summaries, and outcome, so a resume agent sees what happened.

## 15. CLAUDE.md Update at Program End

Bake audit lessons into the project-rules file before the final `JUDGMENT` message.

**What to add (concrete, no fluff):**
| Source | Where in `CLAUDE.md` |
|--------|----------------------|
| Each BLOCKER fix that introduced a new coding rule | New "## Hard Rules" or extension of existing list |
| Each confirmed antipattern (e.g., a `!!` we replaced, a missing `BuildFeatures` guard we added) | New "## Confirmed Antipatterns" section with ID, example, fix |
| Lint config changes (detekt, actionlint) | Inline under "## Workflow Validation" or new "## Quality" |
| New version-catalog / provider-API pattern adopted | Under "## Engineering Principles" |
| Anything that was *not* fixed (deferred, out of scope) | New "## Open Risks" or as inline TODO comments with OWNER-ESCALATION markers |

**What NOT to add:** speculative advice, generic Kotlin tips, any rule that's not anchored to a concrete finding this audit produced.

**Verification:** the diff to `CLAUDE.md` must be surgical — only new sections/bullets, no reflows of existing text. Commit message: `docs(CLAUDE.md): audit lessons (iter N)`.
